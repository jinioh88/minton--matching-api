# 카카오 로그인 Flutter SDK 전환 백엔드 점검

## 배경

Flutter 앱이 카카오 로그인 플로우를 SDK 네이티브 방식으로 전환하면서, 백엔드 `POST /api/auth/oauth/login`의 카카오 처리 검증 요청이 들어왔다.

- Flutter: `com.mintonmatch.app://oauth/callback` 기반 SDK 플로우
- Web: `http://localhost:3000/oauth/callback`(로컬) 기반 OAuth 웹 플로우

---

## 현재 코드 점검 결과

점검 대상:

- `src/main/java/org/app/mintonmatchapi/auth/service/OAuth2Service.java`
- `src/main/resources/application.yml`
- `src/main/resources/application-local.yml`

### 1) 토큰 교환 파라미터

`provider=KAKAO`일 때 토큰 교환 요청은 아래 파라미터를 사용 중이다.

- `grant_type=authorization_code` ✅
- `client_id=<registration.client-id>` ✅
- `redirect_uri=<request.redirectUri>` ✅
- `code=<request.authorizationCode>` ✅
- `client_secret=<registration.client-secret>` ✅(항상 전송)

요청사항과 비교 시, 핵심 파라미터는 일치한다.

### 2) redirectUri 허용 검증

`app.oauth2.allowed-redirect-uris`에 아래 URI가 등록되어 있어 Flutter URI 사용이 가능하다.

- `com.mintonmatch.app://oauth/callback`
- `http://localhost:3000/oauth/callback`
- `http://127.0.0.1:3000/oauth/callback`

즉, Flutter와 Web 콜백 모두 현재 허용 목록에는 포함되어 있다.

### 3) JWT 응답 플로우

카카오 토큰 교환/사용자 정보 조회 성공 후 기존 JWT 발급 흐름은 유지된다.

- 응답 구조: `success`, `data.accessToken`, `data.user`(프로필 완료 여부 포함) 유지 ✅

### 4) 에러 로깅

현재는 전역 예외 처리에서 주로 메시지 중심으로 로깅하고 있으며, 카카오 토큰 API 실패 시 응답 바디(예: `KOE006`, `invalid_grant`)를 구조적으로 남기는 로직은 부족하다.

---

## 수정 필요 항목

### A. `client_secret` 조건부 전송

현재 `client_secret`을 항상 전송하고 있어, 빈 값/미사용 정책 환경에서 불필요한 실패 가능성이 있다.

- 권장: `client_secret`이 비어있지 않을 때만 파라미터에 포함
- 기대 효과: 카카오 설정(사용/미사용) 차이에 대한 호환성 향상

### B. 카카오 토큰 교환 실패 로깅 강화

카카오 API 에러를 분류 가능하게 로그를 남겨야 한다.

- 권장 로그 필드:
  - provider, redirectUri, token endpoint, HTTP status
  - 카카오 에러 바디 원문(`error`, `error_description`)
  - 내부 매핑 에러 코드(예: `KAKAO_INVALID_GRANT`, `KAKAO_KOE006`)
- 예시 구분:
  - `invalid_grant` (인가 코드 만료/재사용)
  - `KOE006` (redirect URI 불일치 계열)

### C. Web/Flutter 경로 분리 문서화

운영/개발 가이드에서 Web과 Flutter를 구분 표기해야 한다.

- Web OAuth 콜백 URI (환경별)
- Flutter 딥링크 콜백 URI
- 카카오 개발자 콘솔에 등록해야 할 Redirect URI 목록
- `POST /api/auth/oauth/login` 호출 예시를 Web/Flutter 각각 분리
