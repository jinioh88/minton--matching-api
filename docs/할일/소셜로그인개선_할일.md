# 소셜 로그인 구조 개선 할일

## 요구사항 분석 요약
- 앱은 소셜 SDK 네이티브 방식으로 로그인하고, 백엔드에는 provider + social access token을 전달한다.
- 백엔드는 소셜사 Me API로 access token을 검증하고 사용자 정보를 조회해야 한다. (Kakao, Google, Naver)
- 웹(React)의 Authorization Code 기반 callback 흐름은 유지해야 한다.
- 로그인 성공 응답은 앱/웹 모두 동일한 서비스 JWT + User 포맷을 반환해야 한다.
- 브릿지 컨트롤러 중심 구조 복잡도를 낮추고 인증 진입점을 단순화해야 한다.

## Step 1 (완료)
- `POST /api/auth/oauth/login`에서 앱 access token 로그인과 웹 authorization code 로그인을 모두 처리하도록 요청/서비스 구조 확장
- Provider별 앱 access token 검증(Me API 호출) 로직 추가 (Kakao, Google, Naver)
- 웹 기존 authorization code 교환 로직 유지, 최종 응답 포맷 단일화

## Step 2 (완료)
- OAuth 브릿지 컨트롤러 정리(`OAuthBridgeController`) 및 관련 보안 허용 경로 정리
- 설정값(`allowed-redirect-uris`) 중 브릿지 의존 항목 정리 여부 점검

## Step 3 (완료)
- 로컬 검증(컴파일/린트) 및 변경 영향 점검
- 앱/웹 로그인 요청 예시 문서화(필드 조합 규칙)
